# 霓虹贪吃蛇网页游戏设计

## 概述

开发一款桌面端优先、可在浏览器运行的经典贪吃蛇游戏，并采用「霓虹深空」未来视觉风格。玩法保持直观：操纵蛇移动、吃食物、不断变长和加速，同时避开墙壁与自身。PixiJS 和 WebGL 负责高品质视觉表现，但不与游戏规则耦合。

首个版本是单人街机游戏，不包含账户系统、后端服务、在线排行榜、道具、障碍或关卡模式。

## 产品目标

- 玩家无需教程即可理解玩法。
- 在普通刷新率和高刷新率显示器上都保持灵敏、顺滑的移动体验。
- 建立鲜明且具有冲击力的未来视觉风格，同时保证网格清晰可读。
- 游戏规则保持确定性，并能脱离渲染层独立测试。
- 优先支持桌面键盘操作，小尺寸浏览器窗口中的画面也能合理缩放。

首个版本需要实体键盘。手机尺寸的屏幕可以正常查看页面，但触控和滑动操作不在本版本范围内。

## 技术方案

应用使用 Vite 和 TypeScript，游戏画面使用 PixiJS 与 WebGL，抬头显示界面和覆盖层使用 HTML/CSS，合成音效使用 Web Audio，最高分及偏好设置使用 `localStorage` 保存。

系统拆分为以下单元：

1. **游戏引擎**：管理网格、蛇、食物、分数、速度、碰撞和状态切换，不依赖渲染或浏览器接口。
2. **输入控制器**：把键盘事件转换为有限长度的方向队列和操作命令，并阻止直接反向移动。
3. **游戏循环**：以固定逻辑节拍推进引擎，并在每一渲染帧中插值显示移动过程。
4. **Pixi 渲染器**：根据引擎快照绘制棋盘、蛇、食物、粒子、拖尾、脉冲和冲击效果。
5. **抬头显示与覆盖层**：通过 HTML/CSS 显示分数、最高分、速度、开始、暂停和游戏结束状态。
6. **音频系统**：在首次用户操作后初始化合成音效，并应用已保存的静音偏好。
7. **持久化适配器**：保存最高分和静音偏好；存储不可用时退化为内存保存。

核心数据流如下：

`键盘事件 → 输入队列 → 固定节拍引擎更新 → 只读快照与游戏事件 → 渲染器、抬头显示和音频`

`foodEaten`、`speedChanged` 和 `gameOver` 等游戏事件用于触发表现效果。渲染器不得修改引擎状态。

## 游戏规则

- 逻辑棋盘为 32 列、24 行，共 768 个格子。
- 桌面端棋盘约为 864×648 个 CSS 像素，并根据可用窗口空间等比缩小。
- 蛇在棋盘中央附近出生，初始朝右，长度为 4 格。
- 在待开始状态按空格键开始游戏。
- 仅使用方向键控制移动，不显示也不支持 WASD。
- 游戏中按空格键暂停或继续；在非游戏中状态按 `R` 重新开始；按 `M` 切换静音。
- 不允许蛇头直接反向撞向第二节身体。
- 输入队列最多保存后续两个有效转向，以接收逻辑节拍之间的快速拐角输入，同时过滤互相冲突的转向。
- 食物均匀随机生成在未被蛇占用的格子中。
- 每吃一个食物，蛇增长一格并获得 10 分。
- 每吃五个食物，速度等级提升一级。
- 移动间隔按公式 `max(65, 150 - floor(foodCount / 5) * 5)` 毫秒计算。
- 撞墙或撞到蛇身时，本局立即结束。
- 蛇的理论最大长度为 768 格。占满全部格子后进入通关状态，不再尝试生成食物。
- 最高分保存在当前浏览器中。

## 状态模型

游戏引擎包含以下明确状态：

- `ready`：显示棋盘和开始覆盖层，按空格键进入游戏。
- `playing`：启用逻辑节拍、输入、渲染、计分和效果。
- `paused`：冻结逻辑时间，按空格键继续，按 `R` 开始新的一局。
- `gameOver`：显示最终分数和重新开始提示，按 `R` 开始新的一局。
- `completed`：蛇已占满棋盘，显示通关效果，按 `R` 开始新的一局。

游戏进行中按 `R` 不生效，避免意外丢失当前进度。游戏进行中失去浏览器焦点时自动进入 `paused`。窗口尺寸变化不会推进或重置游戏状态。

## 视觉与交互设计

已选择的视觉语言为「霓虹深空」：

- 近黑色的深蓝背景，配合低对比度青色数字网格和深空暗角。
- 蛇身采用青色至品红色的发光渐变，蛇头更明亮且易于辨认。
- 食物表现为粉色能量球，并带有克制的同心脉冲。
- 棋盘居中；左上显示分数和最高分，中间显示游戏标识，右上显示速度。
- 游戏框架底部显示紧凑的操作提示。
- 为待开始、暂停、游戏结束、通关及渲染器不可用状态提供响应式覆盖层。

主要效果包括：

- 随速度提升而增强的柔和蛇身拖尾。
- 吃到食物时触发粒子爆裂和网格脉冲。
- 速度升级时触发短暂的等级脉冲。
- 碰撞时触发短促冲击闪光、轻微屏幕震动和克制的故障效果。
- 使用稀疏背景粒子营造空间感，但不得干扰游戏元素。

所有效果必须保持格子边界、蛇身和食物清晰可读。全屏闪光应短暂且保持低透明度。系统开启「减少动态效果」时，减少粒子数量并关闭屏幕震动和故障位移。

## 音频设计

游戏提供合成音效，不使用背景音乐。声音覆盖开始、适当频率的转向、进食、速度升级、暂停和碰撞。为遵守浏览器策略，音频只在首次用户操作后初始化。抬头显示界面提供静音状态，静音偏好保存在浏览器中。

## 性能设计

逻辑更新与画面渲染相互独立。引擎按 150～65 毫秒的配置间隔推进；画面跟随 `requestAnimationFrame` 渲染，并在格子位置之间进行插值。

- 在 120Hz 和 144Hz 显示器上，目标是以显示器原生刷新率渲染；性能足够时达到 120 FPS。
- 在普通设备上，最低目标是稳定保持 60 FPS。
- 粒子使用对象池，几何数据和纹理进行缓存，动画高频路径中避免产生临时对象。
- 限制设备像素比，防止高分辨率屏幕造成过高的填充率开销。
- 持续帧耗时超出目标预算时，自适应质量控制器会减少背景粒子和昂贵的后处理；玩法和前景可读性始终保持不变。
- 文档进入隐藏状态后停止渲染。

实际可见刷新率受到显示器和浏览器限制，不把 60Hz 显示器描述为能够显示 120 个不同画面帧。

## 异常处理

- WebGL 初始化失败时显示「渲染器不可用」覆盖层和重试操作，不显示空白画布。
- `localStorage` 读写发生异常时，在当前会话内使用内存保存分数和偏好。
- 音频初始化失败时静默继续，游戏玩法不受影响。
- 窗口失去焦点时先暂停，再接收新的移动输入。
- 视口小到无法舒适操作时，棋盘仍保持比例，同时显示简短的桌面端建议，不丢弃当前进度。
- 重新开始时创建全新的引擎状态，并清除上一局的临时粒子、输入队列、计时器和覆盖层。

## 测试策略

使用 Vitest 编写确定性的引擎及适配器测试：

- 移动与增长。
- 直接反向拦截和快速转向队列。
- 撞墙与自身碰撞。
- 食物只能生成在空闲格子中。
- 占满棋盘后的通关处理。
- 分数、速度提升及 65 毫秒下限。
- `ready`、`playing`、`paused`、`gameOver`、`completed` 和重新开始状态切换。
- 存储和音频失败时的降级行为。

使用 Playwright 编写浏览器集成与验收测试：

- 仅使用方向键移动。
- 使用空格键开始、暂停和继续，使用 `R` 重新开始，使用 `M` 切换静音。
- 失去可见性或焦点时自动暂停。
- 刷新页面后最高分仍然保留。
- 待开始、暂停、游戏结束和通关覆盖层正确显示。
- 在代表性的桌面和较小视口尺寸下正确布局。
- WebGL 成功启动并输出可见画面。

视觉检查覆盖「霓虹深空」风格、抬头显示层级、特效可读性、减少动态效果模式，以及硬件允许情况下的 60Hz 与高刷新率运行表现。

## 验收标准

- 玩家可以完全通过键盘开始、转向、暂停、继续、失败和重新开始。
- 只有方向键可以控制蛇的移动。
- 32×24 棋盘、增长、计分、加速、碰撞和通关规则符合设计。
- 浏览器存储可用时，最高分和静音偏好能够持久保存。
- 音频或存储不可用时游戏仍可运行；WebGL 不可用时给出明确提示。
- 普通设备在默认效果配置下稳定保持 60 FPS。
- 性能足够的 120Hz 设备以 120 FPS 为目标，并通过自适应质量保护帧时间稳定性。
- 视觉效果不得影响网格、蛇、食物和抬头显示界面的可读性。
- 自动化单元测试和浏览器测试全部通过，生产构建成功完成。
